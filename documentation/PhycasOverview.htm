Phycas Software Design

 

Overview

The Phycas Software consists of two components – a backend processor written in C++ and a front-end GUI written in Java.  The Processor can be controlled through the GUI or accessed directly via the command line.  The Phycas Executor is an additional program which is used to manage the execution of the Processor and the GUI components.

 

This software is based on an XML command language description and schema.  The main XML file is used to auto-generate both C++ and Java code using several XSLT and Python scripts (see Figure 1). There is additional code in both the Processor and the GUI that is hard-coded.

 

Figure 1.  Much of the Java and C++ code is auto-generated from the
command language description XML file.

 

The Processor and the GUI communicate via sockets and can exist on the same machine or different machines.  The Phycas Executor allows the user to choose whether to launch the Processor on the local machine or to connect to an instance already running on a remote machine.  If the local machine is chosen, the user can either enter a specific port number or leave it blank and the Executor will search the localhost for an open port.  If a remote machine is chosen, the user must enter the host ip address and port number.  The host and port number are passed to the GUI and just the port number to the Processor at runtime.  Through the socket connection, the GUI sends commands in Nexus format to the Processor and the Processor sends XML to the GUI (see Figure 2). 

 

Figure 2.   The Phycas Processor and GUI communicate through a Socket connection.  The GUI sends commands in Nexus format to the Processor, and the Processor sends XML to the GUI.

Communication Layer

The Processor will respond to a command from the GUI with XML in one or more of the following tags:

 

  • out
    • Contains output from the processing of the command.
    • GUI displays in Output text tab of Output Window.
  • comment
    • Contains Nexus comments.
    • GUI displays along with data from out tag.
  • error
    • Contains any error messages that may have been generated.
    • GUI displays a dialog box with error message and an OK button.
  • warning
    • Contains any warning messages that may have been generated.
    • GUI displays a dialog box with warning message and an OK button.
  • user_query
    • Contains message that needs to be displayed to the user.
    • file tag contains a message that the Processor found a problem with the input or output file from the command.  GUI displays a dialog box with the message and an OK button so the user can return to the command panel and select a different file.
    • string tag contains a message that needs a variable response from the user.  GUI displays a dialog box with the message, an entry field for the user to type an answer, and an OK button to submit the response.
    • alert tag contains a message to the user that needs no response.  GUI displays a dialog box with the message and an OK button.
    • choices tag contains a message and a list of possible responses.  GUI displays a dialog box with the message, a drop down to choose a response, and an OK button to submit the response.
    • cancel_ok contains a message that needs either ok or cancel as a response.  GUI displays a dialog box with the message and OK and Cancel buttons.
    • no_yes contains a message that needs either yes or no as a response.  GUI displays a dialog box with the message and Yes and No buttons.
    • Processing of tags handled by phycasGUI.swixml.PhycasUserQuery.
  • idle
    • Empty tag signifying that the Processor is now idle.
  • plot
    • Contains data to plot.
    • label tag includes tab-separated string of data labels.
    • entry tag includes a single data entry including a value for each label.
    • Processing of tags handled by phycasGUI.swixml.PhycasChart.
  • hidden_query
    • Response to “available” command, which is sent by the GUI on startup and after each command. 
    • command_state tag contains information on the current state of the program, including which commands and options are available and their current values.  It has a schema similar to the command language but more concise, with additional tags for handling Tax Sets, Tree Sets, and Character Sets.  Each of these set types has a set manager of type phycasGUI.swixml.PhycasSetManager, which contains a group of sets of type phycasGUI.swixml.PhycasSet.  PhycasMain maintains a reference to each of the set managers.  When a command state message is received, PhycasCommandState updates each of the set managers to reflect the currently available sets and their members.

 

GUI Layer

Introduction

phycasGUI.PhycasMainThread is the main class that runs when the Phycas GUI is started.  This class spawns two threads – one for the socket communication called phycasGUI.network.PhycasSocket, and the other for phycasGUI.swixml.PhycasMain which controls the creation of the GUI and interactions with the user.  PhycasSocket maintains the socket connection to the Processor and handles XML messages sent from the Processor.  PhycasMain creates the windows of the GUI and maintains references to them and each of their components.  Most of the GUI components are created using Swixml, a protocol which allows representation of Java Swing components in XML.  These Swixml XML files are auto-generated using the XSLT scripts mainXMLTransform.xsl and commandsXMLTransform.xsl and are rendered upon starting the PhycasMain thread.  The references to the Swixml components must be in the PhycasMain class.  This class is auto-generated using the XSLT script mainJavaTransform.xsl.

 

Handler Classes

PhycasMain also contains references to the Handler classes.  These classes are auto-generated using the Python scripts commandsJavaTransform.py and  cmdParamJavaTransform.py.  The GUI for each command in the command language description is controlled by a class named [Command Label]Handler in the package phycasGUI.swixml.handler.  These classes handle all events generated by the user’s interaction with the GUI for each command, including validation of the data submitted and creation of the Nexus command string.  Another responsibility of the Handler classes is to get and set the data to and from the XMLBeans and the GUI.

 

XMLBeans

XMLBeans are Java objects used to store the data from the initial command language description and the command state updates.  These XMLBeans classes are generated from the XML schemas prior to compilation using the script “scomp”, which generates a jar file of the classes for each schema. 

 

When the PhycasMain thread is started, it parses the command language XML file into a set of command language XMLBeans, package org.phycas.commandLanguage.  When a command state message is received by the GUI, PhycasSocket creates an XML document which phycasGUI.swixml.PhycasCommandState parses into a set of command state XMLBeans, package org.phycas.commandState.  PhycasCommandState then moves the data from the command state beans to the command language beans using reflection (see Figure 3).  Since the two schemas are quite different, some complicated translation between data types must occur.

 

Design

The GUI consists of two windows – the main window and the output window. 

Main Window

When the GUI is initially launched, the welcome screen is displayed in the main window panel.  This screen allows the user to enter a command just as they would via the command-line interface to the Processor.  Each command from the command language description has a menu item in a menu on the menu bar of the main window.  When a menu item is selected, the main window panel displays the appropriate command’s panel.  Each command panel has an OK button.  When this button is pressed, the Nexus command string is generated and sent to the Processor via the socket connection.  The main window panel then switches back to the welcome screen.  If an error occurs in the Processor or more information is needed from the user, the main window panel will switch back to the last command panel viewed.  Additional menu items in the menu bar allow the user to show the output window or the welcome screen, or to exit Phycas.

 

Figure 3.  XMLBeans are created from the command language and the command state schemas.  When a command state message is received by the GUI, the data is moved from the command state XMLBeans to the command language XMLBeans.

 

Output Window

The output window contains a tabbed pane which initially contains a single tab labeled output.  The panel on this tab contains a text area in which all output from the Processor is written.  If a command is issued by the user containing plottable data, tabs will be added to this window to allow the user to generate trace plots of the data returned by the Processor.  These graphs are created by phycasGUI.swixml.PhycasChart which uses the third-party package JFreeChart.  The output window can be closed without exiting Phycas.  The menu bar of this window contains options for saving and clearing the output text, and for printing, saving, closing, and editing the properties of graphs.  Most of these actions can also be performed using a popup menu, which is made visible by right-clicking the mouse in either the output text area or on a graph.  phycasGUI.swixml.PhycasOutput handles all events generated by a user’s interaction with the output window.

JavaHelp System

The main window has a Help menu which activates the JavaHelp system.  JavaHelp provides a mechanism for incorporating a help window into an application, including context-sensitive help.  All of the files necessary for the help system can be found in the gui/help directory.  The Master.hs file is the Helpset file which designates which of the following components will be included:  table of contents, index, search, glossary and favorites.  Note:  Currently, only the first three are implemented.  Master.jhm is the Map file which maps id’s to html files.  This allows for the html files to change names while the application code refers to the same map id.  MasterTOC.xml is the table of contents file which creates the items in the table of contents and sets each of their target id’s.  MasterIndex.xml is the index file which creates the items in the index and sets each of their target id’s.  These target id’s must be found in the Map file.  In order for the search feature to work, a search database must first be created using the command ‘jhindexer’.  This command takes in the directory containing all of the html files as an argument.  Therefore, in order to create the search database, type the command ‘jhindexer master’.  A folder called JavaHelpSearch will then be created which contains the search database files. 

 

All of the help files except for Master.hs are auto-generated using XSLT scripts which are found in the directory gui\help\translate.  The script tocXMLTransform.xsl creates the MasterTOC.xml file.  The script indexXMLTransform.xsl creates the MasterIndex.xml file.  The script mapXMLTransform.xsl creates the Master.jhm file.  The script commandsHTMLTransform.xsl creates the commands.html file in the master directory.  The master directory contains the html files associated with the Master.hs Helpset.  The intro.html file contains introductory information on the software and GUI and is hard-coded.  The commands.html is auto-generated and contains all the help information for all of the commands.

 

The “Launch Help” menu item in the Help menu brings up the Help window initially showing the introduction page.  The “Get help on specific item” menu item allows for context-sensitive help.  When a user selects this item from the Help menu, the cursor will turn into an arrow with a question mark.  The user can then click on a field in the GUI and the help window will be displayed with the help information for that field.  This system is made possible by code in the PhycasMain class which sets id’s for each of the GUI components.  These id’s must be found in the Map file.

 

For more information on the JavaHelp system, see the JavaHelp User Guide.
